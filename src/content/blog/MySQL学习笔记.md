####  1.你先说说，MySQL 里面的 索引 是什么？为什么加索引可以提高查询速度？  
 MySQL 的索引（index）可以理解为数据库表的目录，它不是数据本身，而是额外维护的一种数据结构。 **只有 唯一索引 或 主键索引 才要求唯一。普通索引不要求唯一**。 

比如我们经常根据 email 查询用户，如果没有索引，MySQL 可能要全表扫描；如果给 email 建了索引，MySQL 可以通过 B+ 树快速定位到对应数据，从而减少扫描行数，提高查询性能。

不过索引不是越多越好，因为索引会占用存储空间，而且在 insert、update、delete 时需要维护索引，会带来额外开销。  

```sql
-- 1. 主键索引（PRIMARY KEY）- 自动创建，唯一且非空
CREATE TABLE users (
  id INT PRIMARY KEY,
  name VARCHAR(50)
);

-- 2. 唯一索引（UNIQUE）
CREATE UNIQUE INDEX idx_email ON users(email);

-- 3. 普通索引（INDEX）
CREATE INDEX idx_name ON users(name);

-- 4. 联合索引（复合索引）
CREATE INDEX idx_name_age ON users(name, age);

-- 5. 全文索引（FULLTEXT）
CREATE FULLTEXT INDEX idx_content ON articles(content);
```

#### 2. 为什么 MySQL 的索引通常用 B+ 树，而不是普通二叉树或者哈希表？  
 MySQL InnoDB 的索引常用 B+ 树，主要是因为数据库数据存储在磁盘中，查询性能和磁盘 IO 次数强相关。**B+ 树是多叉平衡树**，**一个节点可以存很多 key，所以树高很低，查找时磁盘 IO 次数少**。相比普通二叉树或红黑树，B+ 树更适合磁盘页存储，普通二叉树每个节点最多两个分支，数据量大时树会很高。如果数据插入顺序不好，还可能退化成链表。  
另外，**B+ 树的叶子节点有序并且通过链表连接**，所以它不仅支持等值查询，还天然支持范围查询、排序和最左前缀匹配。哈希表虽然等值查询快，但不适合范围查询和排序，所以 MySQL 的通用索引结构更适合使用 B+ 树。  

<font style="color:rgb(29, 29, 31);">B+树是自底向上构建的。新数据插入叶子节点，节点满了就分裂，中间值上提到父节点。如果父节点也满了就继续向上分裂，直到根节点。这样能保证树是平衡的，且所有数据都在叶子层，方便范围查询。</font>

#### <font style="color:rgb(29, 29, 31);">3. 什么是聚簇索引和非聚簇索引？InnoDB 里主键索引和普通索引有什么区别？  </font>
InnoDB 中，**主键索引是聚簇索引**，数据行本身就存放在主键索引的叶子节点上，所以通过主键查询可以直接拿到整行数据。  
普通索引是非聚簇索引，它的**叶子节点不存完整数据行，而是存索引字段和对应的主键值**。因此通过普通索引查询时，如果查询的字段不在索引里，就需要先通过普通索引找到主键，再根据主键去主键索引查完整数据，这个过程叫**回表**。  
如果**查询字段都能从普通索引中拿到，就不需要回表，这叫覆盖索引**。  

#### 4.什么是覆盖索引？为什么实际开发中不建议总是 `SELECT *`？  
覆盖索引是指查询需要的字段都可以从某个索引中直接获取，不需要再根据主键回表查询完整数据。比如有联合索引 `(email, username)`，查询 `SELECT id, email, username FROM user WHERE email = ?` 时，由于普通索引里包含 `email`、`username`，并且 InnoDB 普通索引叶子节点会保存主键 `id`，所以可以直接从索引返回结果。  
实际开发中不建议总是 `SELECT *`，因为**它会查询很多不需要的字段，增加磁盘 IO、网络传输和对象映射开销，而且容易破坏覆盖索引，导致回表**。  

#### 5. 什么是联合索引？什么是最左前缀原则？  
联合索引是多个字段共同组成的索引，比如 `(user_id, status, created_at)`。它不是三个独立索引，而是按照字段顺序组织的一棵 B+ 树，先按 `user_id` 排序，`user_id` 相同再按 `status` 排序，最后再按 `created_at` 排序。  
最左前缀原则是指使用联合索引时，查询条件要从索引最左边的字段开始连续匹配。比如索引是 `(user_id, status, created_at)`，那么 `WHERE user_id = ?`、`WHERE user_id = ? AND status = ?` 都可以利用索引；但如果只查 `WHERE status = ?`，因为跳过了最左边的 `user_id`，就不能很好利用这个联合索引。  
实际开发中，我会根据接口的查询条件和排序方式来设计联合索引，比如订单列表经常按 `user_id + status + created_at` 查询，就可以建立对应的联合索引。  

#### 6.如果有一个订单表，常见查询是根据 `user_id` 查询用户订单，并按创建时间倒序分页，你会怎么设计索引？  
```sql
SELECT *
FROM order_info
WHERE user_id = ?
ORDER BY created_at DESC
LIMIT 20;
```

 这个查询是根据 `user_id` 过滤，并按照 `created_at` 倒序取前 20 条，所以我会设计联合索引 `(user_id, created_at)`，如果是 MySQL 8，也可以建成 `(user_id, created_at DESC)`。  
这样 MySQL 可以先通过 `user_id` 定位到某个用户的订单范围，再按照 `created_at` 的索引顺序取数据，减少排序开销。  
我不会建 `(id, created_at)`，因为查询条件没有使用 `id`，不符合最左前缀原则。  

#### 7. 如果这个分页查询存在深分页问题，怎么解决？  
```sql
SELECT *
FROM order_info
WHERE user_id = ?
ORDER BY created_at DESC
LIMIT 100000, 20;
```

`LIMIT 100000, 20` 是**典型的深分页问题。它不是直接定位到第 100001 条，而是通常需要先扫描出前 100020 条数据，然后丢弃前 100000 条，只返回 20 条。**offset 越大，扫描和排序成本越高。如果是 `SELECT *`，还可能带来大量回表。  
优化方式一般是使用游标分页，也叫 seek pagination。比如按 `created_at DESC, id DESC` 排序，下一页把上一页最后一条数据的 `created_at` 和 `id` 作为 cursor，查询比它更早的数据，同时建立 `(user_id, created_at DESC, id DESC)` 联合索引。这样可以避免大 offset 扫描。  

`lastCreatedAt` 不是提前知道的，而是从上一页查询结果的最后一条记录中取出来的。第一次查询不传 cursor，正常按 `created_at DESC, id DESC` 查前 20 条；返回结果时，后端把最后一条记录的 `created_at` 和 `id` 作为 `nextCursor` 返回给前端。下一页请求时，前端再把这个 cursor 传给后端，后端用 `created_at < lastCreatedAt OR created_at = lastCreatedAt AND id < lastId` 继续往后查。这样可以避免 `LIMIT 100000, 20` 这种深分页扫描问题。  

```sql
{
  "list": [
    {
      "id": 10001,
      "createdAt": "2026-06-05 12:00:00"
    },
    {
      "id": 98765,
      "createdAt": "2026-06-01 10:00:00"
    }
  ],
  "nextCursor": {
    "lastCreatedAt": "2026-06-01 10:00:00",
    "lastId": 98765
  }
}
```

#### 8. MySQL 里 `EXPLAIN` 是干什么的？你一般会重点看哪些字段？  
`EXPLAIN` 是用来看 SQL 执行计划的，它不是完整执行过程，而是 MySQL 优化器预估这条 SQL 会怎么执行。实际开发中我会用它分析慢 SQL，比如看有没有走索引、实际用了哪个索引、预计扫描多少行、有没有全表扫描、有没有额外排序或临时表。  
我会重点看 `type`、`key`、`rows` 和 `Extra`。如果 `type` 是 `ALL`，`key` 是 `NULL`，说明可能全表扫描；如果 `Extra` 里有 `Using filesort` 或 `Using temporary`，说明可能存在排序或临时表开销，需要结合索引和 SQL 写法优化。  

```sql
# type从好到差
const > eq_ref > ref > range > index > ALL
# possible_keys 是可能用到的索引。
# key 是 MySQL 最终实际选择的索引。
# rows 是 MySQL 估算要扫描的行数。
# 常见Extra
Using index，Using filesort，Using temporary
```

####  9.如果 `EXPLAIN` 看到 `type = ALL`，一定说明 SQL 有问题吗？  
`EXPLAIN` 里 `type = ALL` 表示全表扫描，但不一定代表 SQL 有问题。要结合具体场景判断。如果**表很小、业务本来就需要全量扫描，或者查询结果占表的大部分数据**，全表扫描可能是可以接受的，甚至可能比走索引更划算。  
但如果是大表上的高频条件查询，比如根据 `email` 查用户、根据 `user_id` 查订单，看到 `type = ALL`、`key = NULL`、`rows` 很大，就说明可能没有命中合适的索引，需要优化 SQL 或索引设计。  

#### 10. 如果线上一个接口突然变慢，你怀疑是 SQL 问题，你会怎么排查？  
 如果线上接口突然变慢，我不会一上来就只看 SQL，而是先通过日志或链路追踪确认耗时主要发生在哪里。如果确认是数据库慢，我会先定位具体慢 SQL，可以通过应用 SQL 日志、APM 或 MySQL 慢查询日志找到。  
然后对慢 SQL 执行 `EXPLAIN`，重点看 `type`、`key`、`rows`、`Extra`，判断有没有全表扫描、有没有命中索引、扫描行数是否过大、是否出现 `Using filesort` 或 `Using temporary`。  
接着结合 SQL 写法和业务场景优化，比如**避免 **`**SELECT ***`**、补充合适的联合索引、处理深分页、避免索引失效、排查 N+1 查询**。如果 SQL 本身没问题，我还会继续看**数据库负载、连接池是否耗尽、锁等待**等问题。  

#### 11.MySQL 事务的四大特性 ACID 分别是什么？  
MySQL 事务的 ACID 分别是**原子性、一致性、隔离性和持久性。**  
原子性是指事务中的操作**要么全部成功，要么全部失败回滚**，比如转账时扣钱和加钱必须同时成功。  
一致性是指**事务执行前后数据库都要处于合法状态**，比如转账前后总金额不应该变化。  
隔离性是指多个事务并发执行时**不能互相干扰**，MySQL 通过锁、MVCC 和隔离级别来保证。  
持久性是指**事务提交后数据要可靠保存**，即使数据库宕机也能恢复，InnoDB 主要依赖 redo log 来保证。  

####  12.MySQL 里有哪些事务隔离级别？它们分别解决什么问题？  
 事务隔离级别是用来控制**多个事务并发执行时，彼此能看到什么数据。**MySQL 常见有 4 个隔离级别：

| 隔离级别 | 介绍 | 脏读 | 不可重复读 | 幻读 |
| :---: | :---: | :---: | :---: | :---: |
| Read Uncommitted |   读未提交隔离性最差，可能读到未提交的数据，所以会出现脏读。     | 可能 | 可能 | 可能 |
| Read Committed |  读已提交可以避免脏读，因为只能读取其他事务已经提交的数据，但同一个事务中多次读取同一行，仍然可能因为别的事务提交修改而结果不同，所以可能出现不可重复读。   | 避免 | 可能 | 可能 |
| Repeatable Read |  可重复读可以保证同一个事务内多次读取同一行数据结果一致。底层主要依赖 MVCC 实现快照读，并通过 Next-Key Lock 在当前读场景下处理幻读问题。   | 避免 | 避免 | 理论可能，InnoDB 做了较多处理 |
| Serializable |  所有事务基本像排队一样执行。  隔离级别最强，但并发性能最差    | 避免 | 避免 | 避免 |


首先看一下三个并发问题：

**1.脏读 Dirty Read  **

** **一个事务读到了另一个事务还没提交的数据。

**2.不可重复读 Non-repeatable Read  ** 

同一个事务里，两次读取同一行数据，结果不一样。例如事务A第一次读数据1000，事务B将其修改为800，第二次事务A再次读就会读成800。

**3.幻读 Phantom Read  **

** **同一个事务里，两次范围查询，第二次多出了或少了符合条件的行。  例如事务A第一次查询发现有10条数据，随后事务B插入一条数据，事务B第二次查询发现11条数据。

MySQL 有四种事务隔离级别：读未提交、读已提交、可重复读和串行化。  
读未提交隔离性最低，一个事务可能读到其他事务还没提交的数据，所以会出现脏读、不可重复读和幻读。  
读已提交只能读到其他事务已经提交的数据，所以解决了脏读，但同一个事务内两次读取同一行，可能因为其他事务提交修改而结果不同，所以还可能出现不可重复读。  
可重复读是 MySQL InnoDB 的默认隔离级别，它可以解决脏读和不可重复读。InnoDB 主要通过 MVCC 和 ReadView 实现快照读，让同一个事务内多次普通查询看到一致的数据版本。  
串行化是最高隔离级别，可以避免脏读、不可重复读和幻读，但并发性能最差，因为事务基本要排队执行。  

#### 13.什么是 MVCC？它解决了什么问题？  
 MVCC （ Multi-Version Concurrency Control  ）是多版本并发控制，是 InnoDB 用来**提高事务并发性能**的一种机制。它的核心思想是给数据保留多个版本，让**不同事务根据自己的 ReadView 读取对应版本的数据**，从而实现**普通读和写之间不互相阻塞**。  
InnoDB 的 MVCC 主要依赖隐藏字段、undo log 和 ReadView。每行数据**有事务 id 和回滚指针**，更新数据时旧版本会记录到 undo log 里，形成版本链。**事务查询时会根据 ReadView 判断当前版本是否可见，如果不可见，就沿着 undo log 找到可见的历史版本。**  
**MVCC 主要用于实现快照读**，比如普通 `SELECT`。它可以避免脏读，并且在可重复读隔离级别下保证同一个事务内多次读取结果一致。但像 `UPDATE`、`DELETE`、`SELECT ... FOR UPDATE` 这类当前读，读的是最新数据，并且通常会加锁。  

#### 14.什么是快照读？什么是当前读？它们有什么区别？  
 快照读是指**普通 **`**SELECT**`** 通过 MVCC 读取一个一致性版本的数据**，它依赖 ReadView，不主动加锁，所以**读写之间通常不互相阻塞**。快照读不一定读取最新数据，可能读取的是历史版本。在 MySQL InnoDB 的可重复读隔离级别下，事务第一次普通查询会生成 ReadView，后续普通查询复用它；在读已提交隔离级别下，每次普通查询都会生成新的 ReadView。  
当前读是读取数据的最新版本，并且通常会加锁，比如 `SELECT ... FOR UPDATE`、`UPDATE`、`DELETE` 都是当前读。当前读适合扣库存、扣余额这类必须基于最新数据处理的场景。  

#### 15. MySQL 里有哪些常见的锁？行锁和表锁有什么区别？  
MySQL 常见锁可以从粒度和模式两方面理解。按粒度分，**有表锁和行锁。表锁是锁整张表，粒度大，并发性能差；行锁是锁具体数据行，粒度小，并发性能好，InnoDB 默认支持行锁。**  
按模式分，有共享锁和排他锁。**共享锁也叫读锁，多个事务可以同时持有，但会阻塞写**；**排他锁也叫写锁，一个事务持有后，其他事务不能再加共享锁或排他锁**。比如 `SELECT ... FOR UPDATE`、`UPDATE`、`DELETE` 都会涉及排他锁。  
InnoDB 还有间隙锁和临键锁。**间隙锁锁的是索引记录之间的范围**例如WHERE id >10 AND id <20，防止其他事务插入 ，**临键锁可以理解为行锁加间隙锁**，它既锁住已有记录，也锁住记录前面的间隙WHERE id BETWEEN 10 AND 20， InnoDB 可能不只锁住 `10` 和 `20` 这两行，还会锁住相关区间，防止其他事务插入新数据造成幻读  ，主要用于可重复读隔离级别下防止幻读。  
实际开发中，比如扣库存、扣余额，一般会通过 `UPDATE ... WHERE stock > 0` 这种原子更新，或者 `SELECT ... FOR UPDATE` 来加行级排他锁，避免并发修改导致数据错误。  

**16. 什么情况下 InnoDB 的行锁可能会退化成表锁，或者锁住比你预期更多的数据？  **

InnoDB 的行锁是基于索引实现的，不是直接锁逻辑上的某一行。如果**查询条件没有命中索引，MySQL 可能会全表扫描，在扫描过程中锁住大量记录，效果上接近锁表**。  
另外，如果是范围查询，比如 `WHERE created_at BETWEEN ...`，并且是 `SELECT FOR UPDATE`、`UPDATE`、`DELETE` 这类当前读，在可重复读隔离级别下可能会加 Next-Key Lock，也就是锁住记录和间隙，防止幻读。**所以范围越大，锁住的数据越多，并发性能越差**。  
实际开发中，我会尽量让更新条件命中索引，避免大范围当前读，事务里也不要做太耗时的操作，减少锁持有时间。

**17. 什么是死锁？MySQL 里死锁一般怎么产生？你会怎么避免？  **

 死锁是指两个或多个事务互相持有对方需要的锁，并且都在等待对方释放，形成循环等待，导致事务都无法继续执行。通常有四个条件：

```sql
互斥：资源一次只能被一个事务持有
占有并等待：事务持有一些锁，同时还在等待其他锁
不可剥夺：锁不能被强制抢走，只能由持有者释放
循环等待：多个事务形成等待环
```

比如事务 A 先更新 `id=1`，再更新 `id=2`；事务 B 先更新 `id=2`，再更新 `id=1`，两个事务加锁顺序相反，就可能出现 A 等 B、B 等 A 的死锁。  
避免死锁的方式主要有：固定加锁顺序，比如统一按 id 从小到大加锁；让 SQL 命中合适索引，减少锁范围；控制事务长度，不要在事务里做远程调用或耗时操作；必要时对死锁异常做有限重试。InnoDB 检测到死锁后会主动回滚其中一个事务，让另一个事务继续执行。  

**18. Spring Boot 里 **`**@Transactional**`** 什么时候会失效？  **

Spring Boot 里的 `@Transactional` 失效通常和 Spring AOP 代理机制有关。比如事务方法不是 `public`，同一个类内部方法调用，没有经过代理对象，事务就可能失效。  
还有一些常见情况，比如异常被 `try-catch` 吃掉没有抛出，Spring 认为方法正常结束，就不会回滚；默认只对 `RuntimeException` 和 `Error` 回滚，如果是受检异常，需要配置 `rollbackFor = Exception.class`；另外，如果类没有交给 Spring 管理、数据库表不是 InnoDB、事务方法里开了新线程，也可能导致事务不生效。  
实际开发中，我会保证事务方法由 Spring Bean 调用，尽量加在 public service 方法上，异常要么继续抛出，要么手动标记回滚，并且常用 `@Transactional(rollbackFor = Exception.class)`。  

```sql
1. 同类内部调用，事务失效
2. 异常被 catch 掉，事务不回滚
3. 默认受检异常不回滚
4. 方法/类没有经过 Spring 代理，事务失效
```

**19.**`**@Transactional**`** 默认遇到什么异常会回滚？如果想让所有异常都回滚，应该怎么写？  **

`@Transactional` 默认**只对 **`**RuntimeException**`** 和 **`**Error**`** 进行回滚**，对普通的受检异常默认不回滚。如果希望所有异常都触发回滚，可以配置 `**@Transactional(rollbackFor = Exception.class)**`。实际开发中我一般会显式加上 `rollbackFor = Exception.class`，避免因为抛出 checked exception 导致事务没有回滚。  

**20. MySQL 里 **`**redo log**`**、**`**undo log**`**、**`**binlog**`** 分别是什么？它们有什么区别？  **

| 日志 | 所属层 | 记录内容 | 主要作用 |
| :---: | :---: | :---: | :---: |
| redo log | InnoDB 引擎层 | 数据页的物理修改 | 崩溃恢复，保证持久性 |
| undo log | InnoDB 引擎层 | 修改前的旧数据 | 回滚，MVCC |
| binlog | MySQL Server 层 | 逻辑变更事件 | 主从复制，数据恢复 |


 redo log、undo log 和 binlog 的作用不同。**redo log 是 InnoDB 的重做日志，记录数据页的物理修改，主要用于崩溃恢复，保证事务持久性**。事务提交后，即使数据页还没刷盘，只要 redo log 落盘，数据库宕机后也能恢复。  
undo log 记录的是**数据修改前的旧版本，主要用于事务回滚和 MVCC**。如果事务失败，可以通过 undo log 恢复原数据；快照读也可以通过 undo log 版本链读取历史版本。  
binlog 是 MySQL Server 层的二进制日志，**记录逻辑变更，主要用于主从复制和数据恢复**。主库写入 binlog，从库读取并重放 binlog 来同步数据。  

**21. MySQL 主从复制大概是怎么工作的？binlog 在里面起什么作用？  **

MySQL **主从复制主要依赖 binlog**。主库执行写操作并提交后，会把数据变更记录到 binlog。从库的 I/O 线程会**连接主库读取 binlog，并写入从库本地的 relay log**；然后从库的 SQL 线程读取 relay log，在从库重放这些变更，从而让从库数据和主库保持一致。  
binlog 在这里就是主从同步的数据来源。实际业务里主从复制常用于读写分离，**写操作走主库，读操作走从库**。但因为主从复制通常是异步的，所以会有主从延迟，**像写后立即读、支付状态、订单详情这类强一致场景一般要读主库。**  

**22. 如果业务用了读写分离，用户刚写入一条数据，马上查询却查不到，你觉得可能是什么原因？怎么解决？**  

如果业务用了读写分离，用户刚写入一条数据马上查询却查不到，很可能是因为写操作走了主库，但读操作被路由到了从库，**而主从复制通常是异步的，从库还没来得及重放主库的 binlog，所以出现主从延迟**。  
解决方式要看业务一致性要求。如果是写后立即读、订单详情、支付状态这类强一致场景，应该强制读主库；如果是普通列表页，可以接受短暂延迟，就可以继续读从库。也可以在**写入后的一小段时间内让该用户的读请求走主库**，或者通过中间件根据请求上下文做主从路由。  

**23. MySQL 中 **`**VARCHAR**`** 和 **`**TEXT**`** 有什么区别？实际开发中你会怎么选择？  **

`VARCHAR` 和 `TEXT` 都可以存字符串，但使用场景不同。`**VARCHAR**`** 适合长度明确、可控的短文本**，比如用户名、邮箱、标题、状态等；`**TEXT**`** 适合存大段文本，比如文章正文、评论内容、聊天消息、模型输出等**。  
实际开发中我会优先用 `VARCHAR`，因为它对索引、查询和字段约束更友好；只有字段长度确实不可控或可能很长时，才使用 `TEXT`。另外，`**TEXT**`** 字段一般不适合频繁作为查询条件或排序字段**，也不建议在列表查询里 `SELECT *` 把大文本都查出来。  

**24. MySQL 里 **`**DATETIME**`** 和 **`**TIMESTAMP**`** 有什么区别？实际开发中你会怎么选择？  **

| **对比** | `**DATETIME**` | `**TIMESTAMP**` |
| :---: | :---: | :---: |
| 是否受时区影响 | 不受时区自动转换影响 | 会受时区影响 |
| 表达含义 | 字面日期时间 | 绝对时间点 |
| 范围 | 更大，约 `1000-9999`<br/> 年 | 较小，常见范围约 `1970-2038`<br/> 或更高，取决于版本 |
| 常见用途 | 业务时间、预约时间、订单时间 | 创建时间、更新时间、日志时间 |


`DATETIME` 和 `TIMESTAMP` 都能存日期时间，但**区别在于 **`**DATETIME**`** 存的是字面上的日期时间，不会根据时区自动转换；**`**TIMESTAMP**`** 表示一个时间点，会受数据库 session 时区影响，存取时可能发生时区转换**。  
`DATETIME` 的时间范围更大，适合存业务时间，比如订单创建时间、预约时间、支付时间。`TIMESTAMP` 常用于记录创建时间、更新时间这类系统时间。  
实际开发中我一般会优先使用 `DATETIME`，并在应用层统一时区，比如统一存 UTC 或统一东八区，避免数据库、后端、前端时区不一致导致时间错乱。  

**25. MySQL 里 **`**count(*)**`**、**`**count(1)**`**、**`**count(column)**`** 有什么区别？ ** 

`COUNT(*)` 和 `COUNT(1)` 都是**统计结果集的行数，包括字段为 **`**NULL**`** 的行**。在 MySQL 中它们通常没有明显性能差异，实际开发中我一般使用 `COUNT(*)`，因为语义最清楚。  
`COUNT(column)` 统计的是**某个字段不为 **`**NULL**`** 的行数**，如果该字段有 `NULL`，这些行不会被计入。所以如果要统计总行数，应该用 `COUNT(*)`，而不是 `COUNT(column)`。  

#### 26. MySQL 里 `DELETE`、`TRUNCATE`、`DROP` 有什么区别？  
| **操作** | **删除内容** | **是否能带 WHERE** | **表结构是否保留** | **常见用途** |
| :---: | :---: | :---: | :---: | :---: |
| `DELETE` | 删除数据行 | 能 | 保留 | 删除部分业务数据 |
| `TRUNCATE` | 清空整表数据 | 不能 | 保留 | 清空临时表、测试表 |
| `DROP` | 删除整张表 | 不能 | 不保留 | 删除不再需要的表 |


`DELETE`、`TRUNCATE`、`DROP` 都和删除有关，但级别不同。`**DELETE**`** 是删除表中的数据行**，可以带 `WHERE` 条件，表结构还在，InnoDB 下可以配合事务回滚。  
`**TRUNCATE**`** 是清空整张表的数据，不能带 **`**WHERE**`，表结构、索引还在，通常比 `DELETE FROM table` 更快，并且**可能重置自增 ID**。  
`**DROP**`** 是删除整张表，数据、表结构、索引都会被删除，执行后表就不存在了**。实际开发中，业务删除一般用 `DELETE`，清空临时表才考虑 `TRUNCATE`，废弃表才用 `DROP`。  

#### 27. MySQL 里内连接 `INNER JOIN`、左连接 `LEFT JOIN` 有什么区别？  
`INNER JOIN` 是**内连接，只返回两张表中满足连接条件的数据**，也就是左右表都匹配上的记录。  
`LEFT JOIN` 是**左连接，会保留左表的全部记录，如果右表能匹配上就返回右表数据，如果匹配不上，右表字段返回 **`**NULL**`。  
比如查询有订单的用户可以用 `INNER JOIN`，查询所有用户及其订单信息，即使用户没有订单也要展示，就用 `LEFT JOIN`。  

**28. 如果一个 **`**LEFT JOIN**`** 后面在 **`**WHERE**`** 里写了右表条件，可能会发生什么问题？  **

`LEFT JOIN` 会保留左表全部数据，但如果在 `WHERE` 条件里写右表字段，比如 `WHERE o.status = 'PAID'`，那么右表没有匹配上的记录会被 `WHERE` 过滤掉，导致 `LEFT JOIN` 的效果接近 `INNER JOIN`。  
如果希望保留左表全部数据，只是限制右表匹配条件，应该把右表条件放到 `ON` 里，比如 `ON u.id = o.user_id AND o.status = 'PAID'`。

#### 29. MySQL 里 `WHERE` 和 `HAVING` 有什么区别？  
`WHERE` 和 `HAVING` 都可以做条件过滤，但执行阶段不同。`**WHERE**`** 是在分组前过滤原始数据行**，不能直接使用聚合函数；`**HAVING**`** 是在 **`**GROUP BY**`** 分组后，对聚合结果进行过滤**，通常会和 `COUNT`、`SUM`、`AVG` 这类聚合函数一起使用。  
比如我要统计每个用户的已支付订单数，并筛选订单数大于 5 的用户，可以先用 `WHERE status = 'PAID'` 过滤已支付订单，再 `GROUP BY user_id`，最后用 `HAVING COUNT(*) > 5` 过滤分组结果。  

#### 30. MySQL 里 `GROUP BY` 是干什么的？如果查询每个用户的订单总金额，SQL 应该怎么写？  
`GROUP BY` 是**按照一个或多个字段把数据分组，通常配合聚合函数使用**，比如 `COUNT`、`SUM`、`AVG`、`MAX`、`MIN`。例如统计每个用户的订单总金额，可以按照 `user_id` 分组，然后对每组的 `amount` 求和：`SELECT user_id, SUM(amount) FROM order_info GROUP BY user_id`。如果要先过滤订单状态，用 `WHERE`；如果要过滤聚合后的结果，用 `HAVING`。  

#### 31. MySQL 里 `UNION` 和 `UNION ALL` 有什么区别？  
| 语法 | 是否去重 | 性能 | 使用场景 |
| :---: | :---: | :---: | :---: |
| `UNION` | 去重 | 较慢 | 需要唯一结果 |
| `UNION ALL` | 不去重 | 较快 | 允许重复，或确定不会重复 |


`UNION` 和 `UNION ALL` 都是把多个查询结果合并成一个结果集。区别是 `UNION` 会去重，而 `UNION ALL` 不会去重。因为 `UNION` 需要额外做去重操作，所以性能通常比 `UNION ALL` 差。实际开发中，如果业务允许重复，或者能确定两个结果集不会重复，我会优先用 `UNION ALL`。  

#### 32. MySQL 里 `IN` 和 `EXISTS` 有什么区别？什么场景下你会用它们？  
`IN` 和 `EXISTS` 都可以用于子查询，但语义不同。`**IN**`** 是判断某个字段的值是否在子查询返回的集合中**，比如 `user_id IN (...)`；`**EXISTS**`** 是判断子查询是否至少能返回一行数据，通常用于关联存在性判断。**  
实际开发中，如果子查询结果集比较小，或者就是明确的集合匹配，可以用 `IN`；如果是判断关联表中是否存在记录，比如用户是否有订单，我更倾向于用 `EXISTS`。另外，`NOT IN` 遇到 `NULL` 容易有坑，反关联场景通常更推荐 `NOT EXISTS`。  

#### 33. MySQL 里什么情况下索引会失效？你能举几个例子吗？  
 MySQL 索引失效常见原因有几类。第一，**联合索引不满足最左前缀原则**，比如索引是 `(user_id, status, created_at)`，但查询只用了 `status`，就不能很好利用这个联合索引。第二，**对索引字段使用函数或计算**，比如 `LOWER(email)`、`age + 1 = 18`，会影响索引使用。第三，`**LIKE**`** 左模糊查询**，比如 `LIKE '%abc'`，普通 B+ 树索引无法从左侧定位。第四，**字段类型不一致导致隐式转换**，比如 `phone` 是 `VARCHAR` 却用数字查询。第五，`**OR**`**、**`**!=**`**、**`**NOT IN**`** 等条件也可能导致优化器不选择索引**。实际排查时我会通过 `EXPLAIN` 看 `key`、`type`、`rows` 和 `Extra` 来确认是否真的命中索引。  

#### 34. MySQL 里为什么不建议在高频查询里使用 `LIKE '%keyword%'`？如果真的要做搜索，你会怎么设计？  
 不建议在高频查询里使用 `LIKE '%keyword%'`，因为**左侧有通配符时，MySQL 的 B+ 树索引通常无法从左边定位，容易导致全表扫描。**  
如果只是**前缀搜索，比如 **`**LIKE 'abc%'**`**，可以给对应字段建普通索引**。  
如果是文章、评论、聊天记录这种全文搜索，我不会直接依赖 MySQL 模糊查询，而会考虑 MySQL FULLTEXT、Elasticsearch 或 OpenSearch 这类**倒排索引**方案。  
如果是 Agent 记忆或文档语义搜索，还可以用 embedding 加向量数据库做语义检索。缓存可以作为辅助，比如把热门关键词结果放到 Redis，但不能只靠前端缓存解决搜索性能问题。  

#### 35. 如果一个表数据量越来越大，比如订单表已经有几千万行，你会怎么做优化？  
 如果订单表达到几千万行，我不会一上来就分表，**而是先看 SQL 和索引是否合理**，比如用户订单列表要有 `(user_id, created_at)` 联合索引，避免 `SELECT *` 和深分页。然后可以考虑**冷热数据分离，把历史订单归档到历史表，或者做读写分离减轻主库压力。**  
如果单表仍然压力很大，再考虑**水平分表。订单表常见做法是按 **`**user_id**`** 哈希分表**，这样同一个用户的订单落到同一张表，查询用户订单列表时可以直接路由到对应表。也**可以按时间分表，适合按月份归档和统计**，但查询用户全部历史订单可能需要跨表。  
我不太会优先按省份这种枚举字段分表，因为容易数据倾斜，而且路由和统计都比较麻烦。按订单 id 范围分表可以用于归档，但新订单可能集中写入最新表，产生热点。
